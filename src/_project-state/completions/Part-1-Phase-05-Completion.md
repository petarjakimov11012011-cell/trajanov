# Part 1 · Phase 05 · Code — Completion Report

**Date:** 2026-07-12 · **Outcome (one line):** a customer can now go cart → order form → confirmation end-to-end — `/cart` (review, change qty, remove, subtotal), `/order` (validated cash-on-delivery form, server-side submit), and `/order/thanks` — with the order-send built as a safe stub (no real email, no secret) that also has a testable failure path.

## 1. What shipped (plain language)
The shop can now take an order. `/cart` (which used to 404) lists what's in the browser cart — name, size, colour, unit price, a −/+ quantity control, a line total and a Remove control — with a running subtotal in MKD and a Proceed-to-order button; an empty cart shows a quiet message and a link back to the catalog. `/order` is the delivery form (full name, phone, address, city — Macedonia only, no country field), states cash-on-delivery plainly, checks each field with a clear inline message, and submits on the server. Submitting is real but the email itself is a **safe stub this phase**: on success the cart empties and the customer lands on `/order/thanks`; on a (test-forced) failure the page stays put, the cart is kept, and a message offers to retry or order via Instagram. No real email is sent, no secret is used, and no new package was installed — the real send drops in at Phase 2.01.

## 3. Decisions I made during this phase
Five decisions recorded in `decisions.md`. The first three were baked into the brief; the last two are executor calls:
- **D-1.05-1** — order form is Macedonia-only: name/phone/address/city, city free text, no country field. Alt rejected = a country dropdown (cross-border ordering is UNVERIFIED). Follow-up = add a country field if confirmed.
- **D-1.05-2** — order email is a stub this phase; no `resend` installed. `sendOrder` makes no network call and uses no secret; the real send is Phase 2.01 at the marked spot.
- **D-1.05-3** — no delivery/shipping line anywhere; the total is the item subtotal only (delivery cost is UNVERIFIED, arranged off-site).
- **D-1.05-4** (executor) — extended the cart store with `setQty` / `removeLine` / `clear` (+ a pure `cartSubtotal` helper), all on the **unchanged** line-item shape, so every mutation writes through the one store and the header badge stays in sync. Alt rejected = mutating localStorage from the page (two writers, badge drift risk).
- **D-1.05-5** (executor) — a cart line stores no image (fixed shape) and product data is server-only, so the cart thumbnail is the shared 3:4 treated placeholder tile, not the product photo (both the tile and the name link to the product via slug). Alt rejected = changing the line-item shape or fetching product data client-side (both out of scope). Follow-up = add an `image` to the line item once the shape can move (post-1.04b).

Minor implementation choices (not logged as `D-` decisions):
- **Submission = a Next.js server action** (`src/app/order/actions.ts`, `"use server"`) wrapping `sendOrder`, rather than a route handler. Either satisfies "server-side"; the server action is the lighter seam for 2.01's secret-bearing call. `order.ts` itself is a plain server module (types + `sendOrder`) so its types can be imported into the client via `import type`.
- **Dev-only payload log** in `sendOrder` (`process.env.NODE_ENV !== "production"`) so the payload is inspectable during verification without logging customer PII in production.
- **Hydration gating + a `placed` flag on `/order`** — the cart/order pages gate their empty state on the store's `hydrated` flag to avoid a flash; the order page sets an internal `placed` ref before clearing the cart on success so the "empty cart → /cart" redirect doesn't hijack the navigation to `/order/thanks` (see §4).

## 4. Deviations from the brief
None in scope or output — all four tasks delivered as specified. One bug was found and fixed **during** in-browser verification (not a deviation, but worth recording): on a successful submit, clearing the cart momentarily made the cart empty, which tripped the order page's "empty cart → redirect to `/cart`" guard and beat the push to `/order/thanks`, landing the user on `/cart`. Fixed by setting a `placed` ref before `clear()` and skipping the redirect when it's set; re-verified that a successful submit now reliably reaches `/order/thanks`.
